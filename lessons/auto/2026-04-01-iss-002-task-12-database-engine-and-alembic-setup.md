# Lessons Learned: Task 1.2: Database Engine and Alembic Setup

## What Worked Well
- Following the established config pattern (importing the Settings singleton from app.config) kept database.py consistent with the rest of the codebase and avoided re-reading env vars directly
- Scoping docker-compose commands to `docker-compose up -d postgres` prevented unrelated frontend failures (missing nginx.conf) from blocking migration testing
- Writing a manual (non-autogenerate) initial migration for extensions and ENUM types gave full control over the upgrade/downgrade lifecycle and produced a clean autogenerate baseline afterward
- Running a quick import smoke-test (`python -c "from app.database import engine"`) before invoking alembic commands surfaced any import errors with clean Python tracebacks rather than cryptic alembic failures

## What Was Challenging
- Configuring alembic's env.py for async engines requires a non-obvious pattern (AsyncEngine.begin() with run_async_migrations) that differs significantly from the synchronous Alembic documentation; the async path is easy to misconfigure silently
- The volume mount (`./backend:/app`) masking Docker build artifacts means the editable install (.egg-info) must exist on the host for alembic to resolve `app.database` imports inside the container
- ENUM types in PostgreSQL are not transactional in the same way as tables; downgrade logic must explicitly DROP TYPE and handle dependency order carefully

## Key Technical Insights
1. Async SQLAlchemy engines require the `postgresql+asyncpg://` URL scheme — a plain `postgresql://` URL will fail with a confusing error rather than a clear "wrong dialect" message. Validate or assert the scheme at engine creation time.
2. asyncpg is pinned to `<0.30` due to potential breaking changes with the SQLAlchemy 2.x async dialect. Do not upgrade either package without re-testing the full async engine connection end-to-end.
3. Alembic's async env.py must use `AsyncEngine.begin()` inside an `asyncio.run()` wrapper for the online migration path; the offline path remains synchronous and uses the URL string directly from alembic.ini or the env var.
4. `CREATE EXTENSION IF NOT EXISTS "uuid-ossp"` and ENUM type creation should be in a manual initial migration, not autogenerated, to ensure idempotency and clean baseline detection on subsequent `--autogenerate` runs.
5. `alembic.ini`'s `sqlalchemy.url` should reference the `DATABASE_URL` environment variable (via `%(DATABASE_URL)s` interpolation or by overriding in env.py) rather than hardcoding credentials.

## Reusable Patterns
- **Async engine pattern**: Create engine with `create_async_engine(settings.database_url, echo=False)`, session factory with `async_sessionmaker(engine, expire_on_commit=False)`, and a `get_db` async generator that yields a session and closes it in a finally block.
- **Alembic async env.py**: Override `run_migrations_online` to use `asyncio.run(run_async_migrations())` where `run_async_migrations` calls `AsyncEngine.begin()` and passes the connection to `context.run_migrations()`.
- **Import smoke-test before alembic**: `python -c "from app.database import engine, async_session, Base, get_db"` — run this before any `alembic upgrade` or `alembic revision` command to catch broken imports early.
- **Isolated postgres testing**: Always use `docker-compose up -d postgres` (not the full stack) during database migration development to avoid coupling test failures to unrelated services.
- **ENUM downgrade safety**: In the downgrade function, drop dependent tables or columns before dropping ENUM types to avoid FK/column dependency errors.

## Files to Review for Similar Tasks
- `backend/app/database.py` — canonical async engine, session factory, Base, and get_db pattern for this project
- `backend/alembic/env.py` — async-compatible Alembic environment configuration
- `backend/alembic/versions/<hash>_initial_extensions_and_enums.py` — reference for manual migration structure with PostgreSQL extensions and ENUM types
- `backend/app/config.py` — pydantic-settings v2 SettingsConfigDict pattern; import Settings singleton from here, never read os.environ directly in application modules

## Gotchas and Pitfalls
- **Wrong URL scheme**: `postgresql://` instead of `postgresql+asyncpg://` causes a confusing SQLAlchemy async dialect error — always validate the scheme at startup.
- **Volume mount masking .egg-info**: The `./backend:/app` mount hides the container's build artifacts. If alembic cannot import `app.database`, check whether the editable install's `.egg-info` directory exists on the host, not just inside the image.
- **asyncpg version lock**: Do not bump asyncpg to `>=0.30` without explicit re-testing; pin to `<0.30` until SQLAlchemy 2.x async compatibility is confirmed for the newer release.
- **Frontend service blocking tests**: Running `docker-compose up -d` without specifying a service will attempt to start the frontend stub, which fails due to missing `nginx.conf`. Always scope to `postgres` during this milestone.
- **ENUM types are not transactional in PostgreSQL**: If a migration fails mid-flight after creating an ENUM type, the type may already exist when retrying — use `CREATE TYPE ... IF NOT EXISTS` where supported, or handle the exception in the migration.
- **Cryptic alembic errors from broken imports**: If env.py has any import error, alembic will surface a confusing top-level error rather than a Python traceback. Always run the import smoke-test first.
