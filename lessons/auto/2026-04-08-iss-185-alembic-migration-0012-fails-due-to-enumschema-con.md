# Lessons Learned: Alembic migration 0012+ fails due to enum/schema conflicts — requires manual workaround

## What Worked Well
- Consolidating ENUM creation into migration 0001 with idempotent guards resolved the recurring clean-database failure pattern
- Using PL/pgSQL `DO $$ BEGIN IF NOT EXISTS ... END $$` blocks eliminated the asyncpg `CREATE TYPE IF NOT EXISTS` incompatibility
- Fixing column name mismatches (count → current_count, missing group_id) by reading ORM models directly against migration DDL caught discrepancies before they surfaced in tests
- Running an import smoke-test (`python -c 'from app.models import *; from app.database import Base'`) before pytest caught broken ORM imports as clean Python tracebacks rather than cryptic Alembic errors
- `DROP TYPE IF EXISTS` in downgrade functions is safe with asyncpg and requires no workaround

## What Was Challenging
- asyncpg silently differs from standard psycopg2 in DDL syntax — `CREATE TYPE IF NOT EXISTS` is a PostgreSQL 9.6+ feature but asyncpg's parser rejects it, making the failure non-obvious
- Alembic autogenerate silently drops `server_default` and `onupdate` directives from timestamp columns — migrations 0012-0017 required manual review and patching
- Stamping `head` after manual table creation skipped migrations 0015 and 0016, meaning JSONB translation columns and survey versioning columns were absent without any error
- The `./backend:/app` volume mount masking container build artifacts (`.egg-info`) caused import failures that looked like Alembic configuration problems

## Key Technical Insights
1. asyncpg does NOT support `CREATE TYPE IF NOT EXISTS` — it throws `PostgresSyntaxError: syntax error at or near 'NOT'`. All idempotent ENUM creation must use the PL/pgSQL catalog check pattern.
2. The correct asyncpg DDL execution method is `conn.exec_driver_sql()`, not `conn.execute(text(...))` — the latter does not bypass SQLAlchemy's statement compilation layer for raw DDL.
3. Alembic autogenerate is unreliable for columns with `server_default`, `onupdate`, or complex PostgreSQL-specific defaults — always manually author or review migrations for timestamp columns.
4. `alembic stamp head` marks the current state as fully migrated without running any migration code — using it after manual DDL silently skips any real migrations that added columns after the table was created.
5. New or modified ORM models must be imported in both `alembic/env.py` AND `app/models/__init__.py` before running any alembic command; missing either causes silent migration gaps.
6. conftest.py must mirror all ENUMs created by migrations — a missing `response_status` ENUM in the test fixture causes test failures that appear unrelated to the migration issue.

## Reusable Patterns
- **Idempotent ENUM creation (asyncpg-safe):**
  ```python
  await conn.exec_driver_sql(
      "DO $$ BEGIN "
      "IF NOT EXISTS (SELECT 1 FROM pg_type WHERE typname = 'enum_name') "
      "THEN CREATE TYPE enum_name AS ENUM ('val1', 'val2'); "
      "END IF; END $$"
  )
  ```
- **Safe ENUM teardown:** `DROP TYPE IF EXISTS enum_name` works fine with asyncpg in downgrade functions
- **Pre-test import smoke-test:** `python -c 'from app.models import *; from app.database import Base'` — run inside the container before pytest to surface ORM import errors cleanly
- **Migration column audit:** diff ORM model `mapped_column` names against migration `sa.Column` names before running tests — catches count/current_count-style mismatches early
- **Scope docker-compose during migration work:** `docker compose up -d postgres` only — avoids backend/frontend container noise during schema debugging

## Gotchas and Pitfalls
- **Never use `CREATE TYPE IF NOT EXISTS` with asyncpg** — it will throw `PostgresSyntaxError` even though the syntax is valid PostgreSQL; always use the PL/pgSQL DO block workaround
- **Never use `conn.execute(text(...))` for raw DDL** — use `conn.exec_driver_sql()` to bypass SQLAlchemy compilation
- **Never trust `alembic stamp head`** to represent a fully-migrated schema — it only updates the version table, it does not run migration code; columns added in later migrations will be absent
- **Never trust autogenerate for timestamp columns** — `server_default=func.now()` and `onupdate` are silently dropped; manually add them after autogenerate
- **Always add new ENUMs to conftest.py** — the test fixture creates ENUMs independently of migrations; a mismatch causes failures that look like application bugs, not schema bugs
- **Check `.egg-info` exists on the host** when using `./backend:/app` volume mounts — the volume mount shadows container build artifacts, breaking editable installs and causing cryptic import errors
